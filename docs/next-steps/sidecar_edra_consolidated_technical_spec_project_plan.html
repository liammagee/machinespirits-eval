Consolidated engineering specification · Sidecar-EDRA · v1.0 · 2026-05-27

Rigorous event-driven adaptation, starting from a locked baseline.

This document consolidates the paper, the dramatic-recognition sidecar, and the EDRA bridge into one implementation plan. The goal is not another prompting experiment. The goal is a reproducible evaluation harness that makes every improvement, regression, and within-run trajectory visible down to the transcript, policy decision, critic vote, and aggregate plot.

01Engineering contract

The next leg must be run as a measurement system, not as a narrative search. EDRA may be added only after the present sidecar has been reproduced under a locked baseline and the harness can show progress and regressions at run, item, turn, and critic levels.

Core claim to test. A policy/device/closure layer can improve event-local adaptation-to-recognition by increasing branch-local pressure handling, strategy-family selection, public device use, and old/new check articulation, without increasing no-cue or routine-control leakage.

02Evidence constraints inherited from the paper and sidecar

Paper constraint: level effects are supported; slope effects are not.

Paper 2.0 supports calibration and error correction, but its primary adaptive-responsiveness analysis is null: tutors change across turns, yet recognition and architecture do not reliably steepen trajectories. Therefore the new system should not optimize for generic slope improvement alone.

  • Keep average trajectory slope as a secondary diagnostic.
  • Promote event-local metrics: trigger detection, strategy shift, habit-break, device use, old/new re-reading.
  • Do not claim a repaired general adaptive-responsiveness mechanism until it survives main-harness tests.

Sidecar constraint: adaptation is visible but closure is unstable.

The sidecar successfully narrows the construct: the target is no longer average score slope but a staged event in which the learner creates pressure, the tutor changes move, and the learner publicly re-reads the earlier difficulty. The latest loop got one full pass and two 6/9 repeats.

  • Retain clean anchors D42/D50/D53 as the baseline termination set.
  • Retain D42/D45-style low-organic controls and D25/D26 hard traps for calibration.
  • Treat action-to-re-reading as the primary engineering bottleneck.
Source resultEngineering interpretationHarness requirement
Recognition narrows output variance and lifts first-turn level.Recognition is calibration, not necessarily adaptation.Baseline plots must separate level metrics from event-local adaptation metrics.
Superego correction is useful mainly where calibration does not pre-empt failures.Critics are safety nets and observability layers, not magic adaptation engines.Log initial response, critique, revision delta, and final response separately.
State-policy variants choose the right move more often; dialogue engine writes better prose.Separate policy selection from prose compilation.EDRA must compile from a structured MoveContract, not let the writer choose its own strategy silently.
v4 branch-local correction reduced false positives.Prefix leakage is a measurement bug, not a subtle success.All scorer and analyzer functions take prefixThroughTurn as a required argument.
Latest loop failures include critic splits, origin ambiguity, insufficient scores, and scorer error.Failure localization needs to be encoded, counted, and plotted.Every failed item gets a primary failure_code plus secondary contributing codes.

03Target architecture: sidecar first, EDRA as a bridge layer

Do not replace the sidecar. Insert EDRA as an event-local policy/device/closure layer around the existing generation and scoring path.

04Baseline protocol: B0 before any EDRA intervention

The first engineering milestone is a frozen baseline of the current sidecar. It should reproduce the current boundary as faithfully as possible and become the denominator for every later change.

B0 acceptance target. A baseline is accepted when it produces a complete immutable run bundle, passes schema validation, has blind scoring coverage above threshold, renders all required plots, and can regenerate aggregate metrics from raw transcripts and score rows without manual intervention. It does not need to “succeed” at recognition; it needs to measure the current system honestly.

B0 run matrix

LayerRequired baseline contentsRationale
Primary scenariosD42, D50, D53These are the latest gated-loop anchors and should be used for direct comparability.
Calibration controlsD42, D45, plus existing flat/trap brackets such as D4, D25, D26, and optionally D10 as boundary trap.Separates low-organic flatness, hard traps, and boundary critic sensitivity.
Armsprefix_baseline, routine, none, peripeteia_only; positive-control reframe_only optional but recommended.Retains the negative and positive calibration surfaces that made the sidecar useful.
RepeatsMinimum 3 iterations for the D42/D50/D53 termination set; minimum 2 draws for broader controls.One pass is not enough; the current observed boundary is repeat stability.
Critic panelFour blind critics, matching the sidecar panel where possible. Store model names and versions in manifest, not prompt context.Preserves 3-of-4 recognition rule and critic-split detection.
TerminationBaseline run terminates after configured repeats, not after success. Do not search for a clean draw.Baseline estimates current behavior; it is not a generation loop to rescue the system.

B0 command surface to implement

# 2. Run current sidecar without EDRA modules.

# 3. Score blindly with the configured critic panel.
npm run sidecar:score -- --run-id B0_current_sidecar --blind --critics qwen,gemini,deepseek,sonnet

# 4. Build item gates, aggregate metrics, and plots from raw artifacts.
npm run sidecar:metrics -- --run-id B0_current_sidecar --out artifacts/metrics/B0_current_sidecar.json
npm run sidecar:dashboard -- --metrics artifacts/metrics/B0_current_sidecar.json --out reports/B0_current_sidecar.html

# 5. Lock the baseline. Later runs may compare to it but must not overwrite it.
npm run sidecar:lock-baseline -- --run-id B0_current_sidecar

07Plotting requirements: progress, deltas, and drill-down

Every experiment should produce a static HTML report with the plots below. The report is not just for presentation; it is the debugging surface for coding agents.

The dashboard should show this kind of within-run progression for every loop, plus confidence intervals and failure-code overlays where applicable.

Required dashboard panels

  1. Run header: run id, baseline id, code commit, prompt hashes, model versions, cost, coverage.
  2. Gate waterfall: generated → scored → branch-local pressure → habit-break → device performed → old check → replacement check → 3-of-4 closure.
  3. Delta vs B0: each metric with absolute delta, relative delta, bootstrap or Wilson interval, and pass/fail status.
  4. Within-run progress: iteration-by-iteration item-gate rate, control leakage, closure, critic splits, and cost.
  5. Scenario-arm heatmap: rows scenarios, columns arms, cells metric values and failure codes.
  6. Critic agreement matrix: pairwise agreement and per-critic permissiveness.
  7. Failure ledger: sortable table of item failures with transcript links and trace links.

Baseline runner

async function runBaseline(configPath: string): Promise<RunMetrics> {
  const config = loadAndValidateConfig(configPath);
  assert(config.mode === "baseline");
  assert(config.edra.enabled === false);

  const manifest = await freezeManifest(config);
  const cases = expandScenarioMatrix(config.scenarios, config.arms, config.repeats);
  const shuffled = deterministicShuffle(cases, config.randomization.seed);

  const transcriptWriter = jsonlWriter(pathFor(manifest.runId, "transcripts.jsonl"));
  const traceWriter = jsonlWriter(pathFor(manifest.runId, "traces.jsonl"));

  for (const scenarioCase of shuffled) {
    validatePrefixThrough(scenarioCase);
    const trace = new TraceRecorder(manifest.runId, scenarioCase.itemId, traceWriter);

    const transcript = await trace.span("generation.current_sidecar", async () => {
      return currentSidecarGenerate(scenarioCase, config.generator);
    });

    transcriptWriter.write({
      runId: manifest.runId,
      itemId: scenarioCase.itemId,
      scenarioId: scenarioCase.scenarioId,
      arm: scenarioCase.arm,
      prefixThroughTurn: scenarioCase.prefixThroughTurn,
      transcript,
      manifestHash: manifest.hash
    });
  }

  const scoreRows = await blindScoreRun(manifest.runId, config.critics);
  const itemGates = await analyzeItemGates({ manifest, scoreRows, prefixStrict: true });
  const metrics = computeRunMetrics({ manifest, scoreRows, itemGates, baseline: undefined });
  await renderDashboard(metrics, { baseline: undefined });
  await lockBaseline(manifest, metrics);
  return metrics;
}

EDRA adaptive turn runner

async function runEdraItem(scenarioCase: ScenarioCase, config: EdraConfig): Promise<TranscriptItem> {
  const trace = new TraceRecorder(config.runId, scenarioCase.itemId);

  const event = await trace.span("event.read", () =>
    EventReader.read({
      transcript: scenarioCase.prefixTranscript,
      latestLearnerTurn: scenarioCase.branchLearnerTurn,
      prefixThroughTurn: scenarioCase.prefixThroughTurn,
      scenarioMetadata: scenarioCase.metadata
    })
  );

  const ledger = await trace.span("state.update", () =>
    StateLedger.update(config.memoryStore, event)
  );

  const policy = await trace.span("policy.select", () =>
    PolicySelector.select({ event, ledger, allowedFamilies: config.allowedMoveFamilies })
  );

  const device = await trace.span("device.retrieve", () =>
    DeviceLibrary.retrieveOrGenerate({ event, policy, scenarioCase })
  );

  const contract = MoveContractBuilder.build({ event, policy, device });
  validateMoveContract(contract);

  const tutorDraft = await trace.span("tutor.compile", () =>
    TutorCompiler.write({
      transcript: scenarioCase.prefixTranscript,
      contract,
      recognitionPrompt: config.prompts.recognitionCompiler
    })
  );

  const closureCheck = await trace.span("closure.critic", () =>
    ClosureContractCritic.check({ contract, tutorDraft, scenarioCase })
  );

  const tutorTurn = closureCheck.pass
    ? tutorDraft
    : await trace.span("tutor.repair_compile", () =>
        TutorCompiler.rewriteWithClosureFailures({ tutorDraft, contract, failures: closureCheck.failures })
      );

  const learnerFinal = await scenarioCase.learner.respondTo(tutorTurn);

  await trace.span("memory.curate", () =>
    EpisodeMemoryCurator.record({ event, contract, tutorTurn, learnerFinal, closureCheck })
  );

  return assembleTranscriptItem({ scenarioCase, event, contract, tutorTurn, learnerFinal, traces: trace.flush() });
}

Metrics and comparison

function computeRunMetrics(input: {
  manifest: RunManifest;
  scoreRows: ScoreRow[];
  itemGates: ItemGateResult[];
  baseline?: RunMetrics;
}): RunMetrics {
  const health = computeMeasurementHealth(input.scoreRows, input.manifest.expectedScoreRows);
  const aggregate = summarizeMetrics(input.itemGates, input.scoreRows);
  const byIteration = groupSummaries(input.itemGates, "iteration");
  const byScenarioArm = groupSummaries(input.itemGates, ["scenarioId", "arm"]);
  const byCritic = computeCriticAgreement(input.scoreRows);

  const deltasVsBaseline = input.baseline
    ? compareAgainstBaseline({ current: aggregate, baseline: input.baseline.aggregate, pairing: "scenario-arm-draw" })
    : undefined;

  const hardGateStatus = evaluateHardGates({ health, aggregate, deltasVsBaseline });

  return {
    runId: input.manifest.runId,
    baselineId: input.baseline?.runId,
    runStartedAt: input.manifest.runStartedAt,
    manifestHash: input.manifest.hash,
    measurementHealth: health,
    aggregate,
    byIteration,
    byScenarioArm,
    byCritic,
    deltasVsBaseline,
    hardGateStatus,
    itemGateRows: pathFor(input.manifest.runId, "item_gates.jsonl"),

10Project plan and milestones

The milestones are ordered so that engineering rigor precedes architectural ambition. Each milestone produces an artifact that later coding agents can verify.

M0

Repo and artifact hygiene

Create canonical directories, schemas, config names, manifest writer, and dashboard output location. Ensure raw transcripts stay out of Git while committed manifests are sufficient to locate archived payloads.

  • Deliverables: schema files, manifest writer, artifact README.
  • Tests: schema validation and archive-path tests.
Gate: local smoke passes
M1

B0 current-sidecar baseline

Run and lock the current sidecar on D42/D50/D53 plus control brackets before EDRA is enabled.

  • Deliverables: B0_current_sidecar manifest, raw archive, metrics JSON, static dashboard.
  • Tests: score coverage, blind scoring, prefix-through accounting, reproducible metrics.
Gate: baseline locked
M2

Instrumentation and plot harness

Add traces for model calls, module calls, scorer calls, metrics aggregation, and plot generation. Every plot point must link to item rows.

  • Deliverables: trace schema, long-form plot data, dashboard drill-down.
  • Tests: no aggregate without source rows; dashboard fixture tests.
Gate: no black boxes
M3

Analyzer hardening

Make branch-local pressure, origin ambiguity, old/new check naming, and failure-code priority deterministic and unit-tested.

  • Deliverables: analyzer package, golden fixtures, mutation tests.
  • Tests: off-by-one prefix fixtures; control-leakage fixtures.
Gate: analyzer stable
M4

A20 EDRA policy compiler without memory

Insert EventReader → PolicySelector → DeviceLibrary → MoveContract → TutorCompiler → ClosureCritic, but keep memory off.

  • Deliverables: module outputs, contract logs, comparison dashboard versus B0.
  • Success: strict shift and public habit-break improve without control leakage.
Gate: policy/prose split tested
M5

Closure bridge repair loop

Use the closure critic to repair tutor turns before learner final response when the device/action/old-new contract is underspecified.

  • Deliverables: repair traces and before/after compiler deltas.
  • Success: old-check and replacement-check rates improve without quality-warning inflation.
Gate: action-to-re-reading lift
M6

Device skill library

Persist reusable devices with trigger conditions, contraindications, closure tests, success examples, and failure examples.

  • Deliverables: skills/pedagogical_devices/*.yaml, retrieval tests.
  • Success: similar triggers retrieve consistent devices and improve held-out anchors.
Gate: reusable devices
M7

Episode memory and cross-run learning

Add outcome-indexed memory only after non-memory EDRA is measured. Memory stores trigger → move → device → closure outcome, not vague learner personality.

  • Deliverables: memory store, curation policy, forgetting/retirement rules.
  • Success: held-out generalization improves; stale hypothesis rate remains low.
Gate: memory earns keep
M8

Regression CI and release discipline

Wire cached eval tests into PRs and scheduled live drift tests into nightly jobs. Prevent prompt/model/scorer updates from merging without comparison dashboard.

  • Deliverables: CI workflow, release checklist, report index.
  • Success: any regression produces a visible report before merge.
Gate: engineering loop closed

11Coding-agent task packet

This section is written as direct implementation guidance for coding agents.

Start here. Do not implement EDRA modules first. Implement manifest/schema/tests, then reproduce B0. Only after B0 is locked should you add policy/compiler/closure modules.

Task queue

  1. Create schema package: implement TypeScript or JSON Schema for RunManifest, ScenarioCase, TraceSpan, ScoreRow, MoveContract, ItemGateResult, RunMetrics.
  2. Add manifest freezer: hash scenario fixtures, prompts, scorer prompts, code commit, model configs, and analysis config.
  3. Implement B0 config: config/edra/a19-b0-current-sidecar.yaml using current sidecar code path with EDRA disabled.
  4. Implement trace writer: JSONL first; optional OpenTelemetry adapter second.
  5. Implement blind scorer normalization: every critic row parsed into the same schema; retries logged; failures counted.
  6. Implement item-gate analyzer: branch-local pressure after prefix-through, public habit-break, device performed, old check named, replacement check named, recognition 3-of-4, failure codes.
  7. Implement metrics builder: aggregate, by iteration, by scenario-arm, by critic, deltas versus B0.
  8. Implement static dashboard: plots plus sortable failure ledger with links to item artifacts.
  9. Run and lock B0.
  10. Only then implement EDRA modules: EventReader, PolicySelector, DeviceLibrary, MoveContractBuilder, TutorCompiler, ClosureContractCritic, FailureLocalizer, MemoryCurator.

12Reference map

These sources motivate the architecture and harness. Use them as design justification, not as substitutes for sidecar-specific evidence.

  1. Paper 2.0 v3.0.106. Mechanism results: calibration and error correction supported; adaptive responsiveness null; sidecar contribution boundary added.
  2. Dramatic recognition sidecar arc, 2026-05-26/27. Clean anchors, gated loop, current action-to-re-reading boundary, archive policy.
