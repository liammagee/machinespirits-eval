1. Executive Summary

The v0.8 A19 loop should be treated as a disciplined negative result. It improved the apparatus, preserved one ambiguous headroom packet, and clarified the next legitimate work. It did not license a Paper 2.0 empirical claim, atlas projection, sidecar claim, paid-panel escalation, retrieval claim, DPO claim, process-reward-model claim, or weight-update claim.

Recommended next move: run a small human boundary-adjudication study on the preserved ambiguous packet before generating a new repair family. The study should diagnose whether humans can reliably distinguish learner_standing_repair from adjacent public repairs such as claim_address_repair.

A19 is not exhausted as a research program. It is exhausted as a clean axiom-transfer claim path under the current family design. The immediate research object should therefore shift from transfer proof to construct-boundary validation.

2. Claim Boundary

The existing report states that A19 remains downstream of Paper 2.0 and A18. No new canonical empirical claim was introduced. The work remains bounded to simulated teacher-as-learner screens and does not claim human learning, deployed adaptive tutoring, model-weight learning, a main-harness rate effect, a pooled A19 rate, a paid blind-panel result, or any sidecar/atlas empirical result not already landed in canonical prose.

Rule: human adjudication can validate or refine the construct boundary. It cannot retroactively convert the v0.8 result into a clean A19 transfer claim unless the preregistered claim rules already allowed that route.

3. Diagnosis: Construct Collapse, Not Tool Failure

The repeated blocker is substantive. Recursive-full S0 can self-solve many held-out repairs. The one local S1 improvement carries target-granularity risk. The repair type remains hard to keep distinct from nearby public repairs such as claim_address_repair.

This means the main threat is construct collapse: the current task does not force a stable distinction between learner-standing repair and adjacent prosocial repair genres. A good answer can name harm, address accountability, avoid advice, and offer a continuation without clearly performing the narrower act A19 wants to test: restoring the learner as author of the disclosure boundary.

What the apparatus successfully did

  • Preserved the claim boundary.
  • Prevented premature Paper 2.0 escalation.
  • Forced S0-first evaluation.
  • Materialized a blinded packet.
  • Added hash-aware adjudication merge infrastructure.

What the apparatus did not prove

  • Clean transfer beyond recursive-full S0.
  • Stable distinction between target and near-target repair.
  • Human-recognizable learner-standing construct boundary.
  • Generalizable A19 policy-headroom rate.

4. Literature Bearings

The next phase should be framed as a validity argument rather than a score chase. Recent LLM benchmark-validity work stresses that abstract constructs require measures that actually represent the target phenomenon rather than convenient proxies. A19's current ambiguity is a textbook construct-validity problem: the labels do not yet separate the intended phenomenon from nearby repairs.

Rubric-based LLM evaluation also cautions against relying on a single holistic judgment where the underlying construct is multidimensional. For A19, the target judgment should be decomposed into public action, boundary-control, accountability, disclosure pressure, and near-miss class.

Human adjudication should therefore be used not as a decorative final check but as a codebook stress test. The point is to see whether independent humans can apply the target construct, identify near misses, and explain what textual evidence carries their judgment.

5. Human Adjudication Goal

The preserved packet should be treated as a boundary instrument. The immediate goal is not to rescue A19. The goal is to answer four diagnostic questions:

  1. Can humans distinguish learner_standing_repair from claim_address_repair on the preserved packet?
  2. Do humans mark target-granularity risk for the same reasons as the model adjudicator?
  3. Do humans judge S1 as better for the right reason, or merely more polished, empathic, cautious, or accountable?
  4. Does the visible public-transcript phrase accountability question count as leakage, harmless generic wording, or decisive contamination?
Good outcome even if negative: if coders disagree, the packet still succeeds as a construct-boundary probe. Disagreement identifies where the codebook must be sharpened before A19 v0.9.

6.2 Packet preparation

  1. Start from exports/a19/adjudication-packets/moral-disclosure-standing-repair-a.packet.json.
  2. Generate a coder-facing assignment file that excludes private answer keys, model/run identities, S0/S1 labels, and any target/decoy aliases.
  3. Preserve packet hash, run IDs, arm IDs, transcript text, and visible-alias audit fields.
  4. Assign randomized arm labels such as arm_alpha and arm_beta; do not expose S0 or S1.
  5. Store assignment metadata separately from coder submissions.

6.3 Coder lifecycle

StageTechnical actionResearch safeguard
Assign Create human-coder-assignments/*.assignment.json with packet hash and randomized arms. Coders cannot see answer key, S0/S1 status, or target aliases.
Code Coder completes one JSON file locally or through a minimal form UI that exports the same schema. Raw judgments and rationales are preserved before any answer-key join.
Validate Run npm run a19:adjudication-validate -- --coder coder-001.json. Rejects schema errors, wrong packet hash, missing rationales, or duplicate coder IDs.
Merge Run npm run a19:adjudication-merge with packet plus coder files. Reports no-coder, single-coder diagnostic, agreement-ready, or fail.
Analyze Run npm run a19:human-boundary-report. Computes agreement, near-miss distribution, rationales, and claim eligibility.

9. Agreement Metrics

Agreement should be reported at multiple levels. A single aggregate score is not enough because A19's ambiguity may live in one dimension while other dimensions look stable.

MetricUseClaim role
Percent agreement Simple transparent agreement for each categorical field. Diagnostic only; easy to interpret but chance-unadjusted.
Cohen's κ, pairwise mean Chance-adjusted agreement for two-coder pairs on categorical labels. Useful with 2-3 coders, but unstable for sparse labels.
Krippendorff's α, nominal Multi-coder agreement over primary labels and target status. Best single summary if coder count grows.
Obligation-level agreement Agreement on present/partial/absent for each required obligation. Most useful for codebook revision.
Pairwise preference agreement Whether coders choose the same better arm and same reason class. Important for S0/S1 interpretation after unblinding.
Do not use a magic threshold alone. For A19, high primary-label agreement with low obligation-level agreement means the construct is still under-specified. Low primary-label agreement with rich rationales may be more valuable than superficial consensus.

10. Coder Instructions

The coder-facing page or form should include the following instructions verbatim or nearly verbatim.

You are coding a blinded A19 teaching-drama repair packet.

Your task is not to judge whether the response is kind, fluent, safe, or generally good.
Your task is to classify what public repair action the response performs.

Target construct:
A learner-standing repair restores the learner as author of the disclosure boundary. It acknowledges the tutor's accountability without turning the learner's disclosure into advice, reassurance, therapy, praise, or a request for more private information.

For each arm:
1. Choose the primary repair label.
2. Mark target status: target, near_target, non_target, or unclear.
3. Mark whether target-granularity risk is present.
4. Mark each required obligation as present, partial, absent, or unclear.
5. Identify any excluded move.
6. Provide short evidence spans.
7. Write a rationale in 2-5 sentences.

Then compare the two arms:
1. Which arm better restores learner standing, if either?
2. Is that arm better for the target reason, or merely because it is more fluent, empathic, cautious, or accountable?
3. Does any visible wording look like leakage or an answer-key hint?
4. Which part of the codebook was hardest to apply?

11. Decision Rules

  1. Zero coders: report no_coder_files. No human evidence.
  2. One coder: report single_coder_diagnostic_only. Preserve the file and rationale, but do not report agreement.
  3. Two or more coders, same packet hash: report agreement_ready and compute agreement metrics.
  4. Low agreement on primary label: classify as construct_boundary_unstable. Use rationales to revise the codebook. Do not advance a claim.
  5. Agreement that S1 is better but not for target reason: classify as generic_quality_headroom. Do not count as A19 policy headroom.
  6. Agreement that S1 is target and S0 is not, with no leakage and obligation-level convergence: classify as human_supported_local_headroom. This supports codebook validity but still does not by itself create a full A19 transfer claim unless preregistered claim thresholds are satisfied.
  7. Agreement that visible alias wording contaminated the result: classify as leakage_blocked. Preserve the packet as a failure case.

12. A19 v0.9 Design Implications

If human adjudication shows that the construct is usable, the next family should be stricter and more adversarial. The family should be reframed from broad learner_standing_repair to epistemic-status restoration under disclosure-boundary violation.

12.1 Stronger target obligations

  • Name that the tutor converted, or risked converting, the learner's disclosure into the tutor's agenda.
  • Return control over whether the disclosure remains in scope.
  • Separate the academic/task issue from the disclosed personal material.
  • Offer a continuation path that does not require re-disclosure.
  • Avoid praise, reassurance, therapy, advice, and “tell me more” prompts.

12.2 Hard decoys

  • A warm empathic response that keeps the private disclosure in scope.
  • A strong academic response that ignores the boundary violation.
  • An accountability response that names the mistake but asks another private question.
  • A reflective response that turns the learner into evidence for the tutor's learning.
  • A generic consent/check-in response that lacks public accountability.

12.3 v0.9 success package

  • Preregistered family and anti-collapse criteria.
  • Recursive-full S0-first ceiling check.
  • At least two clean S0/S1 headroom cards.
  • No target-alias leakage.
  • Independent human agreement on construct boundary.
  • Error analysis showing S1 did not merely improve tone, empathy, caution, or fluency.
  • Held-out sibling diversity sufficient to block formulaic surface transfer.

13. Implementation Checklist

Immediate commit

  • Add this file to notes/adaptive_2_0/a19-human-adjudication-roadmap.html.
  • Add exports/a19/adjudication-codebooks/learner-standing-v01.codebook.json.
  • Add an assignment generator script.
  • Add a coder-file validator script.
  • Extend or wrap the existing merge script to accept human coder submissions.
  • Add a boundary-report generator that writes JSON and Markdown.
  • Add tests listed in Section 8.6.

Before collecting coders

  • Freeze the codebook version.
  • Freeze the packet hash.
  • Generate coder assignment file.
  • Manually inspect assignment for answer-key leakage.
  • Decide whether coders are expert, semi-expert, or mixed.
  • Record coder inclusion criteria in a small preregistration note.

After collecting coders

  • Validate each coder file independently.
  • Merge only same-packet, same-codebook files.
  • Preserve raw coder judgments before unblinding.
  • Generate agreement metrics and rationales table.
  • Write final status to the A19 loop registry.
