Curriculum Objects Become Worlds

A practical path from an authored curriculum graph to a locked adaptive-world contract, then to a light teaching drama run.

The curriculum object is the stable source of truth. The world adaptation spec is the locked runtime contract. The drama file is the lightweight enactment layer. Keep those three artifacts separate and the system can plan, teach, and inspect without turning a generated rubric into its own evidence.

01·

Artifact map

There are three layers, and each has a different job.

a.

Curriculum object source

curriculum/ai-foundations.curriculum.yaml is the CASE-inspired course graph: modules, knowledge components, tasks, verifiers, misconception signatures, prerequisites, and delivery metadata.

b.

World adaptation spec contract

curriculum/ai-foundations.worlds.yaml contains locked world_adaptation_spec records. Each spec gives the adaptive tutor local evidence, permitted action families, expected transitions, forbidden moves, and outcome observability for one curriculum module.

c.

Drama spec enactment

curriculum/ai-foundations.mvp-dramas.yaml turns curriculum modules into teachable scenes for scripts/generate-pedagogical-dramas.js. This is where a tutor, learner, director, persona, and turn plan become a transcript.

Boundary. The world spec can constrain and annotate action. It is not the evaluator. Success still comes from learner turns, verifier evidence, transcript quality, or an independent judge.

Rule Compile first, then run. Do not let a live dialogue invent its own world contract.

02·

Build the object

A curriculum object is a graph plus machine-checkable extensions.

The portable spine is close to 1EdTech CASE: a curriculum document, module items, knowledge-component items, and associations. Machine Spirits adds fields that adaptive teaching needs: canonical tasks, verifiers, misconception signatures, mastery gates, transfer challenges, and drama/world bindings. 

schema_version: ms-curriculum-v0.1
id: ai_foundations_v1
version: 1.0.0
standard_profile:
  spine: 1EdTech CASE 1.1 inspired
  extensions:
    - ms:evidence
    - ms:verifier
    - ms:misconception
    - ms:drama_binding
    - ms:world_adaptation
modules:
  - id: AF6
    title: Evaluation, generalization, calibration, and shift
    main_artifact: model audit and claim-evidence table
    primary_verifier: metric engine and evaluation linter
    knowledge_components:
      - id: AF-601
        statement: Construct and interpret a confusion matrix
    canonical_tasks:
      - Derive metrics from counts rather than memorized formulas
    verifiers:
      - A metric engine recomputes all values from predictions...
    misconception_signatures:
      - High accuracy means a good classifier

To regenerate the AI Foundations object from the markdown source: 

npm run curriculum:convert:ai-foundations

A runnable module must have verifier evidence and misconception evidence. If either is absent, the world compiler should fail rather than create an underspecified runtime contract.

AF12 The capstone uses non-standard prose, so the parser canonicalizes capstone deliverables into tasks, verifiers, and misconception signatures.

03·

Compile the world

The world spec is the module's locked adaptation contract.

The world compiler lowers each selected module into one world_adaptation_spec. Its hash is deterministic and content-sensitive, so a trace can later prove which world contract governed the run. 

npm run curriculum:compile:worlds
npm run curriculum:compile:worlds -- --check
world_adaptation_specs:
  - id: W_AF6_CURRICULUM
    version: ms-world-adaptation-v0.1
    source_curriculum_id: ai_foundations_v1
    module_id: AF6
    locked_at_compile_time: true
    spec_hash: sha256:...
    learner_state_evidence:
      knowledge_components: [...]
      misconception_signatures: [...]
      verifier_signals: [...]
    action_policy:
      allowed_action_families:
        - diagnose_with_discriminating_question
        - request_evidence
        - contrast_models
      preferred_action_families:
        - repair_overconfidence
        - request_evidence
      disallowed_action_families:
        - model_worked_example
    expected_transitions:
      - action_type: request_evidence
        success_evidence:
          - learner-authored rationale
        failure_evidence:
          - mere agreement
    forbidden_moves:
      - id: no_hidden_label_exposure
        move: hidden_label_exposure
    outcome_observability:
      artifact: model audit and claim-evidence table
      primary_verifier: metric engine and evaluation linter
      independent_outcome_required: true
      spec_is_not_evaluator: true

The compiler currently defaults to the MVP slice: AF1, AF4, AF5, AF6, AF11, and AF12. Use --all only when every module is ready for runtime constraints.

Hash The trace stores the spec id and hash, not a floating prompt description.

04·

Operational semantics

Inside the adaptive tutor, the world constrains action, not truth.

The Plan 2.1 closed loop already estimates learner state, selects a finite action, validates an adaptation contract, realizes a tutor utterance, observes the learner's next turn, and closes the intervention ledger. The world spec enters as adaptivePolicyConfig.world_adaptation_spec.

1. Estimate stateBelief over learner hypotheses from observable dialogue.
2. Select actionAllowed/preferred/disallowed world action families filter and rank candidates.
3. Validate contractThe gate rejects disallowed world actions and hidden-label/proof-supply shortcuts.
4. Realize utteranceTutor-visible text receives the action, not the hidden curriculum answer key.
5. Observe outcomeSuccess requires learner evidence; the world spec alone does not satisfy it.

A profile or scenario can resolve a world spec from the generated file: 

runner: adaptive
scenario_source: config/my-curriculum-scenarios.yaml
adaptive:
  architecture: state_policy_closed_loop
  world_adaptation_source: curriculum/ai-foundations.worlds.yaml
  policy:
    mode: closed_loop

Then each scenario chooses the world by id or by curriculum module: 

scenarios:
  - id: af6_metric_claim_light
    curriculum_module_id: AF6
    opening: I think high accuracy is enough to say the classifier is good.
    max_turns: 3

No leak Misconception signatures are runtime evidence categories, not learner-facing labels.

05·

Light run

A light drama run uses the drama spec for enactment and the world spec for adaptation constraints.

First compile the MVP drama file. This creates teachable scenes from the same curriculum object and carries each runnable module's locked world spec id/hash plus a public-safe constraint projection into curriculum_binding. 

npm run curriculum:compile:drama -- --mvp --out curriculum/ai-foundations.mvp-dramas.yaml

Then dry-run one curriculum drama. This checks routing, shuffling, persona, and turn-plan shape without writing files or calling an LLM. 

node scripts/generate-pedagogical-dramas.js \
  --dry-run \
  --spec curriculum/ai-foundations.mvp-dramas.yaml \
  --only D_AF6_CURRICULUM \
  --max-turns 2

For a free transcript artifact, run the mock path into ignored exports. This is the first operational smoke after the dry-run. 

node scripts/generate-pedagogical-dramas.js \
  --mock \
  --force \
  --spec curriculum/ai-foundations.mvp-dramas.yaml \
  --only D_AF6_CURRICULUM \
  --max-turns 2 \
  --out-dir exports/curriculum-light-drama/sample \
  --transcripts-dir exports/curriculum-light-drama/transcripts \
  --delib-dir exports/curriculum-light-drama/deliberation \
  --writing-pad-dir exports/curriculum-light-drama/writing-pad \
  --key exports/curriculum-light-drama/key.yaml

The held-out key and trace now preserve world_adaptation.spec_id and world_adaptation.spec_hash. The generator injects only the public-safe artifact, verifier, action-policy, success-observable, and forbidden-move projection into private tutor/learner constraints; hidden misconception IDs, answer keys, and hashes stay out of public transcript text.

Cost posture Use --dry-run first, --mock second, and only then an attended real generator.

06·

Scenario authoring

The builder hands off to a third graph when a module becomes a reasoning mystery.

Keep three graphs distinct. The curriculum prerequisite DAG orders modules. The compiled world adaptation contract constrains tutor actions. A staged scenario proof DAG connects public rules and released evidence to an answer. The Curriculum Builder reports the first and compiles the second; it deliberately does not invent the third.

1. Align the questionThe question, formal answer pattern, secret, and mirror must ask for the same semantic type.
2. Audit each clueThe learner-facing surface must warrant its exact formal fact; access, motive, or species identity cannot silently become action or attribution.
3. Minimise the proofEvery declared path breaks when any required premise is removed; corroboration, mirrors, controls, and alternate witnesses are labelled rather than treated as extra proof steps.
4. Write public lawEvery rule has an ordinary-language domain gloss that explains why its premises license its conclusion.
5. Set costume and statusContemporary/speculative worlds declare ecology, diction, ledger term, and summary; screen and smoke worlds declare non-production eligibility.
# Full structural catalog gate; no model call or database access
npm run derivation:quality

# Detailed human authoring contract
curriculum/SCENARIO-DAG-GUIDE.md

The static gate checks plot lint, rule glosses, minimal and complete proof paths, duplicate-fact alternatives, non-progress evidence roles, mirror incompatibility, presentation, and eligibility. It cannot certify scientific or legal truth: surface-to-fact alignment and domain entailment still require human review.

Builder handoff Every *.builder-report.md now carries this gate and points to the authoring guide.

07·

Acceptance checks

A curriculum-world-drama slice is only ready when these checks pass.

Canonical parse

The curriculum object validates and includes module tasks, verifiers, and misconception signatures.

World hash

Every compiled world spec has a deterministic spec_hash and fails validation if edited.

Policy constraint

The adaptive selector respects allowed, preferred, and disallowed action families.

Gate fallback

The proof/release gate rejects a disallowed world action even if supplied manually.

Light drama dry-run

The generated MVP drama spec can plan a one-drama dry-run with no writes and no LLM call.

Scenario proof quality

Any derivation world passes npm run derivation:quality and its evidence-to-fact links receive human domain review.

Adaptive smoke

A small AF6 adaptive scenario should run in mock mode with world_adaptation_source.

The working invariant is simple: curriculum chooses the world, the world constrains the action, and the learner's next move supplies or fails the evidence. Plan 2.4 boundary
