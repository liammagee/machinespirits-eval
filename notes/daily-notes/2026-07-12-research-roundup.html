Prompt Engineering

MRMS: A Multi-Resolution Memory Substrate for Long-Lived AI Agents UNBLOCK

Jizhizi Li, Amy Shi-Nash (NxtLab Innovations) · arXiv 2607.04617 · July 6, 2026 · https://arxiv.org/abs/2607.04617

Argues that continuity for long-lived agents can't come from just extending the prompt window: an agent must preserve prior experience, retrieve it selectively, distinguish personal context from external evidence, and revise memory as the situation changes. Proposes a memory substrate organized along a representational axis (structured records / vector representations / graph relations) and a temporal axis (short/medium/long-term), with structured records governing eligibility, vectors supporting recall, and graph relations adjudicating support, contradiction, and supersession before gated context projection.

Project relevance: A natural continuation of last week's DRIFTLENS entry (2607.02374, memory-injected reasoning drift) for cell 21's dynamic prompt rewriting (Writing Pad) and cells 48-49's prompt erosion mechanism — both accumulate context into the tutor's prompt turn over turn today with no explicit contradiction/supersession layer. MRMS's "structured records govern eligibility" design is a concrete pattern for services/evalSignature.js's prompt_content_hash drift detection: instead of only flagging that content changed, a structured-record layer could distinguish a benign Writing Pad update from one that contradicts an earlier commitment the tutor made — closer to what "prompt erosion" is actually trying to measure.

Agent Architecture

Governed Individuation: Cryptographically Decoupling an Agent's Learning from Its Authority UNBLOCK

Xue Qin, Simin Luan, Cong Yang, Zhijun Li (Harbin Institute of Technology / Soochow University) · arXiv 2607.04613 · July 6, 2026 · https://arxiv.org/abs/2607.04613

Asks whether an agent that keeps adapting in the field is still confined to what its operator authorized. Binds the agent at boot to a cryptographically frozen identity digest and routes every action through a gate defined over the semantic effect of the action rather than its declared name/intent — so confinement is an invariant of the execution architecture, not a probabilistic training outcome. No amount of self-modification widens authority without an operator-signed identity change. (A companion paper the same day, arXiv 2607.05463 "Governable Individuals," applies the same idea to embodied agents.)

Project relevance: Directly parallels the design problem behind services/adaptiveTutor/'s programmatic constraints and counterfactual replay (cells 110-125): both are attempts to stop an LLM-driven policy from quietly expanding what it does turn over turn. This paper's gate-on-semantic-effect-not-declared-intent framing is a sharper version of what policyActions.js's state-policy constraints are already reaching for, and is worth reading against the P2.2 state-schema ablations (cells 115-123) as a check on whether those constraints are enforced on the *effect* of a tutor/learner turn or just its surface label.

Prismata: Confining Cross-Site Prompt Injection in Web Agents WATCH

Corban Villa, Alp Eren Ozdarendeli, Sijun Tan, Raluca Ada Popa (UC Berkeley) · arXiv 2607.08147 · July 9, 2026 · https://arxiv.org/abs/2607.08147

Web agents interpret natural language as instructions, so third-party or user-generated page content can hijack them via prompt injection — the same trusted/untrusted mixing problem that made Cross-Site Scripting dangerous. Prismata enforces contextual least privilege, deriving a task-specific security policy that constrains both what the agent sees and what it can do, even though that policy has to be derived from page structure entangled with the attacker's own content.

Project relevance: A trust-boundary framing worth holding against how learner-authored content flows into the tutor's own prompt — the dialectical_suspicious superego (cells 22-33, 93-100) and the Writing Pad (cell 21) both fold prior turns, including learner text, back into what the ego reads next. Prismata's "constrain by what the agent is allowed to see, not just what it's told to do" principle is a candidate lens for npm run audit:message-chain and npm run provenance:validate: today they check integrity of the chain, not whether a given turn's content should have had standing to influence a later turn at all.
