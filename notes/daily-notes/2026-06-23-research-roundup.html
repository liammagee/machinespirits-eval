Prompt Engineering

FAPO: Fully Autonomous Prompt Optimization of Multi-Step LLM Pipelines UNBLOCK

Paul Kassianik, Baturay Saglam, Huaibo Zhao, Blaine Nelson, Supriti Vijay, Aman Priyanshu, Amin Karbasi · Cisco Foundation AI · arXiv 2606.19605 · June 2026 · https://arxiv.org/abs/2606.19605

FAPO treats prompt optimization as a three-level escalation problem: it first tries targeted prompt-text edits, then parameter adjustments, and only restructures chain topology when step-level failure attribution identifies a structural bottleneck. An LLM-driven (Claude Code) diagnostic loop evaluates each variant against a score function, proposes scoped changes, and validates them without human re-prompting. On 18 model-benchmark pairs it outperformed the prior GEPA baseline by +14.1 pp on average (15 of 18 wins); code is open-sourced under Apache 2.0.

Project relevance: The three-level escalation maps directly onto the project's prompt-lab autotune workflow (npm run prompt-lab -- autotune), which currently operates at prompt-text level only. FAPO's step-level failure attribution is the missing mechanism for deciding when to escalate from per-prompt tuning to structural changes in the ego/superego dialogue engine — precisely the question that stalled the cells 93–100 superego-variant ablations. The attribution logic could be adapted to route tutor_first_turn_score residuals back to the specific ego or superego prompt that caused them.

Agent Architecture

A Technical Taxonomy of LLM Agent Communication Protocols UNBLOCK

Linus Sander, Habtom Kahsay Gidey, Alexander Lenz, Alois Knoll · Technische Universität München · arXiv 2606.19135 · June 2026 · https://arxiv.org/abs/2606.19135

The paper classifies LLM agent communication protocols along four axes: message semantics, synchrony, topology, and trust model. It surveys the fragmented landscape of emerging standards (MCP, A2A, ACP, and others) and argues that the absence of a shared taxonomy is the primary barrier to reliable composition of multi-agent systems — an interoperability problem that grows worse as protocol proliferation accelerates.

Project relevance: The bilateral ego-superego architecture is itself an asymmetric two-agent protocol: tutor-ego sends a draft, superego returns a critique, ego revises. Formalising this exchange using the taxonomy's "message semantics" axis would make the deliberation loop unit-testable at the wire level and clarify which cells with multi_agent_tutor: true but superego: null are genuinely single-agent vs. self-reflection loops. The topology axis is also directly applicable to the LangGraph adaptive runner (cells 110–113), whose state transitions are currently implicit in services/adaptiveTutor/graph.js.

Phoenix: Safe GitHub Issue Resolution via Multi-Agent LLMs WATCH

arXiv 2606.20243 · June 2026 · https://arxiv.org/abs/2606.20243

Phoenix decomposes GitHub issue resolution across six specialised agents — Planner, Reproducer, Coder, Tester, Failure Analyst, and PR Agent — coordinated by a label-based GitHub webhook state machine. The "safe" framing comes from explicit rollback hooks and a validation gate that prevents the PR Agent from opening a pull request until the Tester returns a green run.

Project relevance: The Failure Analyst → Coder → Tester feedback loop mirrors the superego → ego-revision → validation cycle in the tutor engine. The label-based state machine is architecturally cleaner than the implicit promise-chain in services/adaptiveTutor/graph.js; worth reviewing if the adaptive runner's state transitions are refactored (TODO B* code quality items).

Pedagogical AI & Tutoring

Learning to Prompt: Improving Student Engagement with Adaptive LLM-based High-School Tutoring UNBLOCK

Po-Chin Chang, Nicholas Hogan, Aske Plaat, Michiel T. van der Meer · arXiv 2606.20138 · June 2026 · https://arxiv.org/abs/2606.20138

The paper formulates adaptive tutoring as a prompt-routing problem: 14 pedagogical features (tutor scaffolding level, student understanding signals, topic switch indicators, etc.) are extracted turn-by-turn from raw transcripts and fed to a lightweight routing model that selects among subject-aware prompt variants. The router is trained offline in simulation then deployed online; an A/B test with 656 conversations from 359 high-school students confirms sim-to-real transfer, with the system switching from analytical to scaffolding strategies in response to student engagement signals (router 0.694 vs. static baselines 0.647 / 0.64, p < 0.001).

Project relevance: The 14-feature turn-level featurizer is a lightweight proxy for rubric v2.2 evaluation: instead of an LLM judge call per turn, a trained routing model could select the next prompt variant in real time — making the adaptive runner (cells 110–113) cheaper to iterate on during prompt-lab autotune sessions. The sim-to-real transfer result also directly supports the external validity argument for the human-learner pilot (§A1 TODO): if simulation-trained routers transfer in this setting, the project's simulated-learner training runs are more predictive of real-learner outcomes than the current conservative framing allows.
