# `notes/daily-notes/` — cadence & naming convention

Research-roundup notes are generated by the deep-research harness (the
`/deep-research` skill / daily-roundup task). This file is the convention the
generator and any human editor follow so the folder stays a clean,
non-overlapping timeline instead of a pile of synonym-named, overlapping-window
digests.

## The drift this prevents

The first two notes showed the failure mode:

| file | framing | window | papers |
|---|---|---|---|
| `2026-06-08-research-digest.html` | "weekly digest" | ~week ending 06-08 | 8 papers |
| `2026-06-09-research-roundup.html` | "roundup, 2–9 Jun" | ~week ending 06-09 | 9 papers |

Two synonyms (*digest* / *roundup*), two **overlapping** weekly windows a day
apart, and — only by luck — **zero shared arxiv IDs**. Run daily, this
accumulates near-duplicate windows fast and makes "what's new since I last
looked?" unanswerable.

## Rules

1. **One type word: `research-roundup`.** Filename grammar is
   `YYYY-MM-DD-research-roundup.html`, date = generation date. Retire *digest*
   as a free synonym. A periodic *aggregate* is the only exception and must be
   named `YYYY-MM-DD-research-digest-<weekly|monthly>.html` **and** list the
   roundup dates it rolls up — an aggregate may overlap roundups only by
   explicit reference, never by re-deriving them.

2. **Non-overlapping windows (delta rule).** Each roundup covers
   `(previous roundup's end, today]`. Windows **tile**: no gaps, no overlaps.
   State the window in a machine-readable header comment (rule 4).

3. **Dedup by arxiv ID (hard rule).** No arxiv ID appears in two roundups.
   Before publishing, compute the IDs already covered in the last 30 days and
   exclude them from the new note:

   ```bash
   # arxiv IDs already covered in the last 30 days — exclude these from a new roundup
   find notes/daily-notes -name '*research-roundup*.html' -mtime -30 \
     -exec grep -ohE '26[0-9]{2}\.[0-9]{4,5}' {} + | sort -u
   ```

   A paper genuinely worth re-surfacing (major revision, new relevance) goes in
   the new note **as a back-reference** ("revisits 2606.xxxxx, first noted
   YYYY-MM-DD"), not as a fresh entry.

4. **Machine-readable header.** First line inside `<body>` (or an HTML comment
   in `<head>`) carries the provenance the rules above depend on:

   ```html
   <!-- meta: date=2026-06-09 window=(2026-06-08,2026-06-09] arxiv=2606.00820,2606.01736,... generator=deep-research/claude-sonnet-4-6 -->
   ```

5. **The folder is the index.** Sorted by filename date, windows tile the
   timeline with no gaps or overlaps. No separate index file to keep in sync.

## Why markdown here, HTML for the notes

The roundups themselves are HTML against the shared techne framework (they are
*read* as documents). This README is an in-repo operational convention — like
`CLAUDE.md` — so it stays markdown.
